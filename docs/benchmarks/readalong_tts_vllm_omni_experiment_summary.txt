Read-Along TTS Experiment Summary
Date: 2026-06-23

Goal
Find a read-along TTS configuration that can generate Qwen3-TTS audio faster than playback while keeping VRAM safely below the 16 GB GPU cap. The read-along app needs stable per-sentence/per-quote highlighting, a short initial buffer, and enough speed headroom for user playback rates above 1.0x.

Important conclusion
The old assumption that character count alone was the main VRAM bottleneck was wrong. The stable performance lever was the vLLM-Omni runtime profile: stage memory reservation, max sequence concurrency, and CUDA graph capture sizes. Lowering the text/model length by itself barely reduced VRAM.

Stack that worked
- WSL Ubuntu 24.04
- Python venv: /opt/ebook-vllm-omni-venv/bin/python
- Backend: vLLM-Omni
- Model: Qwen/Qwen3-TTS-12Hz-1.7B-Base
- Attention path observed in logs: FlashAttention 2
- Read-along profile: scripts/vllm_omni_qwen3_tts_16gb_len8192_seq2_s0util026.yaml
- Stage 0: max_num_seqs 2, max_model_len 4096, gpu_memory_utilization 0.26, CUDA graph sizes [1, 2]
- Stage 1: max_num_seqs 2, max_model_len 8192, gpu_memory_utilization 0.18, CUDA graph sizes [1, 2]
- Time buffer default: 20 seconds
- Playback speed UI range: 1.0x to 4.0x

Why this stack works
vLLM-Omni keeps the two Qwen3-TTS stages resident instead of reloading the model per request. It also uses FlashAttention and CUDA graph capture for the small batch sizes read-along needs. The measured stable profile reduced concurrency from seq4 to seq2, which saved several GB of VRAM while still generating much faster than playback.

Short segment data
The confirmation sweep did include a short, nearly segment-by-segment case:
- 122 characters
- 21 words
- 1 unit
- Generation time: 2.097 s
- Playback time at 1.0x: 8.240 s
- RTF: 0.254
- Smooth speed ceiling: 3.93x

This means precise one-unit generation is feasible for normal playback and likely works for many cases, but it has less headroom at 4.0x than small multi-unit windows. The current read-along default stays "balanced", which generates up to 2 units per request while still returning separate audio items for natural per-unit highlighting. The user can choose "precise" when debugging attribution or voice fidelity.

Why the read-along pipeline is structured this way
1. The app highlights by read-along unit, not by a huge pre-rendered chapter chunk. This keeps sentence/quote sync simple and makes wrong-role/wrong-voice bugs easier to isolate.
2. The backend is resident vLLM-Omni rather than per-call model reload. Startup is expensive, but once resident, individual and small-batch generations are fast enough for buffered playback.
3. The buffer is time-based, not a fixed number of sentences. Sentence lengths vary too much; 2 very short sentences are not the same safety margin as 2 long sentences.
4. The default buffer target is 20 seconds. This gives enough margin for short segments, occasional slow generations, and user playback speed changes up to 4.0x.
5. The default generation mode is balanced. It keeps highlighting accurate because audio is still returned per unit, but it gives better throughput than forcing every unit into a separate model call.
6. The old audiobook generator can likely be sped up by mirroring this vLLM-Omni backend, but that should be a separate integration step. Audiobook generation can use larger windows or seq4/offline profiles because it does not need live UI buffer behavior.

Current recommendation
Use scripts/vllm_omni_qwen3_tts_16gb_len8192_seq2_s0util026.yaml as the default read-along profile on the 16 GB GPU. Treat the lower 10.67 GB seq2/util0.18 result as an experimental lower bound, not a production setting.

Future tests

- Test the Qwen3-TTS 25Hz model through the same vLLM-Omni harness after the 12Hz read-along backend is stable.
- Benchmark a full live read-along session at 4.0x with balanced mode and a 20-second buffer.
